# План A/B-теста

## Цель

Сравнить текущую модель `v1` с кандидатной моделью `v2` для прогнозирования дефолта по кредитной карте.

## Сравниваемые версии

- Контроль: `v1`, `RandomForestClassifier`, текущая модель по умолчанию.
- Тест: `v2`, `GradientBoostingClassifier`, кандидатная модель.
- Текущие offline-метрики:
  - `v1`: F1 `0.5422`, precision `0.5065`, recall `0.5833`, ROC-AUC `0.7734`.
  - `v2`: F1 `0.4688`, precision `0.6634`, recall `0.3625`, ROC-AUC `0.7784`.

## Разделение трафика

API поддерживает два режима:

- Явный выбор через поле JSON `model_version` или HTTP-заголовок `X-Model-Version`.
- A/B-назначение по `customer_id`: сервис считает SHA-256 hash и распределяет клиентов 50/50 между `v1` и `v2`. Один и тот же клиент стабильно попадает в одну группу.

Рекомендуемая длительность теста: 2-4 недели или до набора заранее рассчитанного размера выборки.

## Метрики

- Основная ML-метрика: F1-score для класса дефолта.
- Guardrail-метрика: recall для класса дефолта, чтобы новая модель не пропускала существенно больше рискованных клиентов.
- Бизнес-метрика 1: снижение ожидаемых потерь, рассчитываемых как `probability_of_default * exposure_at_default * loss_given_default`.
- Бизнес-метрика 2: доля одобренных заявок при фиксированном лимите риска.

## Статистический анализ

Для бинарных бизнес-исходов, например дефолт/не дефолт или одобрено/отклонено, используется z-test для двух долей. Для F1 и recall используются bootstrap confidence intervals, потому что эти метрики являются нелинейными функциями элементов confusion matrix.

Кандидатная модель считается успешной, если:

- F1 улучшается, а 95% доверительный интервал разницы не включает ноль.
- Recall не опускается ниже согласованного guardrail.
- Ожидаемые потери ниже или approval rate выше при том же уровне риска.
- Нет деградации p95 latency и доли ошибок API.

## Логирование и мониторинг

Каждый запрос `/predict` получает `request_id` и логируется в JSON Lines с полями `model_alias`, `model_version`, `model_artifact`, `assignment`, `prediction`, `probability_default` и `duration_ms`. Сырые признаки клиента не сохраняются. В production такие логи можно отправлять в ELK/OpenSearch и затем объединять с фактическими исходами погашения для delayed evaluation.

## Влияние на архитектуру

Текущий монолитный Flask API достаточен для учебного A/B-теста: выбор модели происходит внутри одного сервиса, а версии моделей лежат рядом как артефакты. При росте нагрузки можно вынести batch-предсказания и сбор событий в очередь RabbitMQ, а метрики эксперимента считать отдельным consumer-сервисом.
